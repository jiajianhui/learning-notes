# Codex 使用指南

## 1. Codex 是什么

把 Codex 当成一个能直接参与开发的工程师搭档：

- 你负责给目标、限制、优先级
- 我负责分析、实现、检查、解释和迭代

最常见的使用方式不是命令，而是直接对话。

## 2. 我能做什么

`代码工作`
- 阅读项目结构和代码逻辑
- 定位 bug 根因并直接修复
- 做重构、补测试、补脚本
- 跑检查命令并汇报结果
- 做代码 review，重点找 bug、回归和测试缺口

`工程协作`
- 给方案、做拆解、讲取舍
- 帮你推进一个任务直到可用
- 写说明文档、README、变更说明

`文档和效率`
- 把模糊需求整理成可执行任务
- 给出排障步骤、命令清单、使用建议

## 3. 我不适合做什么

- 代替你完成需要账号授权的外部网页操作
- 对最新在线信息做不经核实的回答
- 未经确认执行高风险操作，比如大范围删除、重置

## 4. 最省事的对话方式

直接输入文本就行，不需要每次都用 `/` 开头。

例如：

```txt
帮我看下这个报错
```

```txt
修复 src/app.js 里的问题
```

```txt
先别改代码，先分析根因
```

## 5. 最推荐的提问模板

`排查问题`
```txt
目标：
现象：
复现步骤：
报错信息：
限制：
请先分析，还是直接改：
```

`开发功能`
```txt
目标：
输入/输出：
边界条件：
不想改动的部分：
验收标准：
```

`做 review`
```txt
请按严重程度 review 这次改动，优先找 bug、回归和测试缺口。
```

## 6. 和我协作的高效原则

- 先说清目标和验收标准，效率最高
- 报错优先贴第一条，定位最快
- 可以直接加限制条件  
  `不要改接口`、`不要引入依赖`、`只改这两个文件`
- 赶时间就说  
  `先做最小可用版本`
- 想稳一点就说  
  `先不要改代码，先给方案`

## 7. 典型工作流

1. 你说目标、问题、限制
2. 我分析上下文并确认实现路径
3. 我直接改代码或给方案
4. 我做检查并告诉你结果
5. 你继续反馈，我继续迭代

## 8. `codex xxx` 和 `/xxx` 的区别

这是最容易混的地方。

`CLI 主命令`
- 你在终端最外层直接输入
- 形式是 `codex xxx`
- 作用是启动某个功能

`交互式命令`
- 你已经进入 `codex` 会话后再输入
- 形式是 `/xxx`
- 作用是调整当前会话

例子：

- `codex review`：从终端直接发起一次 review
- `/review`：在当前会话里触发 review 功能
- `codex resume`：从终端恢复旧会话
- `/model`：在当前会话里切换模型

## 13. 权限边界怎么理解

不是“只能看当前文件夹”，而是：

- `cwd` 决定我默认在哪个项目里工作
- `sandbox` 决定我还能读哪里、写哪里

这次会话里，我能读取的不只是一层当前目录，还包括沙箱允许的配置和状态文件。  
但可写目录是受限制的，不是所有地方都能改。

你可以简单理解为：

- 能读的范围通常比能写的范围更大
- 风险更高的操作通常需要额外审批

## 14. 一句话总结

平时直接自然语言对话就够了。  
`codex xxx` 是启动功能，`/xxx` 是调整当前会话。  
不确定时先直接说需求，只有在你明确知道要切模型、切模式、看状态、做 review 时，再用命令。

## 15. Codex CLI 和 Claude Code 是什么关系

可以把它们理解成同一类工具：

- 都是终端里的 AI 编程助手
- 都支持自然语言驱动开发
- 都能读代码、改代码、跑命令、做 review
- 都有命令系统和权限控制

但它们不是“完全一样，只差几个小功能”。

更准确的理解是：

- 操作习惯很像，迁移成本不高
- 设计重点不完全一样

你可以这样记：

- Codex CLI 更像“统一的 agent 平台”
- Claude Code 更像“终端优先的 coding agent 工具”

也就是说：

- 如果你会用其中一个，再学另一个会快很多
- 但不要假设两个工具的命令、模式、权限细节完全一致

## 16. 权限控制到底是什么

权限控制的核心问题只有一个：

**Codex 现在被允许做到哪一步。**

它通常涉及三件事：

- 能不能读文件
- 能不能改文件
- 能不能执行命令

再往下就是更细的限制：

- 能改哪些目录
- 哪些命令可以直接跑
- 哪些操作必须先征求你同意
- 能不能联网

## 17. 为什么要有权限控制

因为 AI 编程助手不只是“聊天”，它真的可能去动你的项目：

- 读取代码
- 修改文件
- 运行测试
- 安装依赖
- 执行 shell 命令

如果没有权限控制，风险会非常高。

所以权限控制本质上是：

**在“效率”和“安全”之间做边界管理。**

## 18. 最容易理解的权限模型

你可以把权限分成 3 层来理解：

`第 1 层：读`
- 看代码
- 看配置
- 看日志

`第 2 层：写`
- 改文件
- 新建文件
- 更新配置

`第 3 层：执行`
- 跑命令
- 跑测试
- 启动服务
- 调用外部工具

一般来说，越往后风险越高。

## 19. 再具体一点：常见限制长什么样

实际使用时，你通常会遇到这些限制：

`目录限制`
- 只能改当前项目目录
- 不能随便改你电脑上其他位置的文件

`命令限制`
- 有些安全命令可以直接跑
- 有些高风险命令需要你批准

`网络限制`
- 有的会话能联网
- 有的会话不能联网

`审批限制`
- 普通操作直接执行
- 敏感操作先问你

## 20. 结合这次会话，怎么理解权限

这次你看到的情况就是一个很好例子：

- 我能读取当前项目目录
- 我也能读取 `~/.codex` 下的一些配置和状态文件
- 但我不能随便写所有目录
- 我实际改动的文件，主要还是在你的项目目录里

所以不要把权限理解成：

“只能看当前文件夹”

更准确的理解应该是：

“默认在当前项目工作，但还会受到沙箱规则、可读范围、可写范围、审批规则的共同约束”

## 21. 你平时怎么判断权限够不够

最简单的判断方法：

- 能看但不能改：说明有读权限，没有对应写权限
- 命令能准备执行，但要先问你：说明触发了审批机制
- 明确报不能访问目录或不能执行：说明碰到了沙箱边界

你不需要把这些机制全记住，只需要知道一件事：

**如果我权限不够，我会告诉你是“读不到、写不了，还是需要你批准”。**

## 22. Codex CLI 和 Codex App 的关系

它们不是简单的“一个有 UI，一个没 UI”。

更准确的理解是：

- Codex CLI 更像本地工作台
- Codex App 更像多 agent 指挥台

CLI 更适合：

- 你在终端里盯着一个任务持续推进
- 边改代码边跑命令边检查
- 和当前项目高频来回协作

App 更适合：

- 同时管理多个 agent
- 跨多个项目并行推进
- 更方便地看线程、历史、状态、任务分工
- 管理更长时间、更偏后台的任务

一句话记：

- CLI = 干活
- App = 调度和管理

## 23. CLI 的对话记录会不会消失

不会。  
至少从你这台机器当前环境来看，CLI 会话记录是会落盘保存的。

我这边已经确认到这些本地记录文件：

- `~/.codex/sessions/`
- `~/.codex/history.jsonl`
- `~/.codex/session_index.jsonl`

另外，`codex resume --help` 也明确说明了它支持恢复之前的交互会话。

所以：

- 关掉 CLI 后，对话记录不会直接消失
- 你之后可以继续恢复之前的会话

## 24. CLI 的记录怎么查看

最直接的方式有两种：

`方式 1：用 Codex 自己看`
- `codex resume`
- `codex resume --last`

这是最推荐的方法，因为它会按会话恢复上下文。

`方式 2：看本地文件`
- `~/.codex/sessions/` 里有会话文件
- `~/.codex/history.jsonl` 里有历史输入记录

如果你只是想“继续上次工作”，优先用 `codex resume`。

## 25. CLI 和 App 的记录会不会同步

根据你这边的实际使用结果，**CLI 对话记录会同步到 Codex App**。

所以现在可以这样理解：

- CLI 会话会保存在本机
- CLI 记录也会出现在 Codex App 里
- 关掉 CLI 不代表会话消失

更实用的结论是：

- 想在终端里继续工作，用 `codex resume`
- 想用图形界面查看线程、历史和状态，可以去 Codex App

也就是说，CLI 和 App 不是两套完全割裂的对话系统，至少在你当前这套使用环境里，历史记录是能接上的。

## 26. CLI 和 App 怎么配合用

- 正在做代码时优先 CLI
- 想看全局状态、历史线程、并行任务时看 App
- 两边不是切换工具，而是在切换工作视角

继续用下去，你通常会越来越明显地感受到：

- CLI 更像手
- App 更像眼
